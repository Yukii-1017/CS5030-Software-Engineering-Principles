Accessibility Compliance


One of the main accessibility compliance is Web Content Accessibility Guidelines (WCAG) 2.1. There are also other compliances like Section 508, which is a federal law in America, and another one The American with Disabilities Acts (ADA). You can refer to 508, ADA, WCAG: What's the difference? to know the difference between them.


Moreover, there are other informal guidelines, like WebAIM - Web Accessibility for Designers, Standards & Best Practices - Technical Standards, Inclusivity, Gestalt Principles, and Plain Language in Document Design and Tips and How Tos | Web Access.


All those restrictions and guidelines are boring and not easy to obey, so we might need other methods to realise web accessibility and get more experience. So the following is some useful advice to improve the accessibility of your website.

Implementation Advice


Firstly, using a formal code is the foundation of website accessibility. For instance, screen readers might follow a document tree to describe the website, so it should be well organized. According to Laura Kalbag (2017), the implementation advice is divided into four aspects: visual impairments, auditory impairments, Motor Limitations and Cognitive Limitations and Mental Disorders.


For Visual Impairments

Following are three different aspects of visual impairments, and we give methods respectively.

  1. Colour Blindness
    The colour of the website should be carefully chosen, which is important for colour blind people to distinguish the content of the website.
    CheckMyColours is recommended because it could give feedback in real-time by inputting URLs. There are also other tools like Colour Contrast Analyser and Color Oracle which need to be downloaded; and other two tools Colour Contrast Check and Luminosity Colour Contrast Ratio Analyser, which could only be used by colour code.

  2. Weak Eyesight
    Weak eyesight people usually look forward to font-size adjust function, which could be realized by CSS. You can refer to Visual Disabilities - Introduction.

    Moreover, Helvetica, Courier, Arial, Verdana and CMU are visual impairments friendly font-style, which means they could lower the burden of readers, as well as dyslexia people. On the opposite, italic is not. (Rello and Baeza-Yates, 2013)

  3. Blindness and Eyesight Loss
    For people who lose the most of eyesight will need voice verification code.

    On the other side, there have been many screen readers in the market, including JAWS Screen Reader, Open Book, Adobe Reader Accessibility, ZoomText, Kurzweil, Apple Voiceover and WebAnywhere. So, the code must be written following the guideline of W3C. For example, add alt to every picture on your website, as they are meaningless for people who can not see it.

    Using VoiceOver to Evaluate Web Accessibility gives a guideline about how to use voiceover(a free system embedded screen reader on Apple’s devices) to test the accessibility of your website.

    Likewise, some people prefer to use a keyboard, you can refer to How to do keyboard testing | Web Access to improve keyboard accessibility.

For Auditory Impairments

There are two aspects of auditory impairments deserve to be considered.

Firstly, If you have audios or videos on your website, you might consider adding plug software to add descriptions or subtitles for them. This will make videos more friendly for those who have hearing limitations. Following are two links that give rich resources related to captions: Auditory Disabilities - Introduction, Captioning Resources. Besides, How to Caption Videos and About Audio Description gives various methods about how to add captions and descriptions.

Then, for some people, gesture language is their first language, there could be a gesture dictionary for them to check.


For Motor Limitations

In terms of motor limitations, there are different attitudes of motor limitation. Some lead to a tremble when using keyboards, touch screens and mice, others might lead to a function loss of muscle so that alternatives like eye trackers or speech-recognition software are used to be a supplement.

At this website, you can find the introduction of different assistive technologies for motor limitations and we suggest developers check the functionality of those functions on their websites.


For Cognitive Limitations and Mental Disorders

Many people with cognitive limitations struggle with understanding ability issues. You should consider to remove the formal nouns of your website or add an explanation. Readability Test Tool is a web where you can test the suitable age of your users. If you want it to suit all different ages, you might expect the result to be as low as possible.

Moreover, you may need to remove pictures which might stimulate people with mental disorders, if there is a higher possibility that they will visit this website, like government or official websites.

General Evaluation Tools


Evaluation tools are used to check the accessibility of websites. It could be divided into two types, subjective assessments and conformance testing. (Zhang, 2018, p.11)


For the first one, one of the common methods is to do questionnaire surveys to website users. The best way to know the thoughts of the users should be testing with real people. In terms of the last one, it includes a checklist and composite testing tools.


In this section, we mainly focus on the analysis of conformance testing methods.

  1. Official W3C Checklist is a set of systematic merits. Its attributes are that it divided all content into “POUR”, four parts, which is an abbreviation of “perceivable, operable, understandable and robust”. In terms of every sub-merit, it gives them different levels from A to AAA, which refers to low website accessibility to high website accessibility. Its advantage is formal and wide-covered, in turn, it means it's hard to use because it would take a long time to check all the restrictions one by one. WebAIM's WCAG 2 Checklist is another checklist website but same as the W3C version except with a different appearance.

  2. There is also a simplified version of the above checklist: Easy Checks – A First Review of Web Accessibility.

  3. An important foundation of website accessibility is the correctness of codes, which could also be checked by W3C at W3C Markup Validation Service, W3C CSS Validation Service or W3C Link Checker. Because the attributes like headers and labels will affect the navigation ability of users, especially people who use screen readers and people with intellectual or motor limitations.

  4. You might also want to know ARIA, which is a supplement of HTML used to improve website accessibility. You can find the rules of the newest ARIA at W3C website. WebAIM and MDN web docs also illustrate good examples of ARIA.

  5. WAVE Web Accessibility Evaluation Tool, Tenon.io | Home page, Siteimprove Accessibility Checker, Run FAE: Functional Accessibility Evaluator 2.1 and Accessify.com are compositive testing tools. They could give detailed information by uploading an URL or raw codes. But Siteimprove and Run FAE are not as effective as the first two, because I prefer others which give results in real-time. In terms of Accessify.com, it could not run a test on my website so not recommended.

  6. There are several Chrome extensions like WAVE Evaluation Tool, axe - Web Accessibility Testing, and Tenon Check. By comparison, I recommend WAVE Evaluation Tool, because it does not need to register compared with the axe, and Tenon’s extension is not accessible. Moreover, there is also one for Safari, 11yTools - Web Accessibility, and another for IE and Opera, WAT, you need download to use WAT.

  7. On the other side, you should also take care of the accessibility of files on the website. Alternate media made easy - Request is a website that allows you to transfer files into other formats. Improve accessibility with the Accessibility Checker - Office Support is a function that is easy to use if you work on Microsoft series products, including office, PowerPoint etc. How to Create Accessible Word Documents | Web Access, How to Create Accessible PDF Documents | Web Access and Guidelines for accessible surveys and forms | UCOP also provide very useful ideas.
